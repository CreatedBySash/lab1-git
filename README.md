## Контрольні запитання з Git

### 1. Основи Git

**Яка різниця між Git та GitHub?**  
Git — це система контролю версій (VCS), яка працює локально. GitHub — онлайн-платформа для зберігання Git-репозиторіїв та спільної роботи над ними.

**Що означає "розподілена" система контролю версій?**  
Кожен користувач має повну копію репозиторію, включно з історією змін. Можна працювати автономно, а потім синхронізувати зміни.

**Яка різниця між git init та git clone?**  
`git init` — створює новий порожній репозиторій локально.  
`git clone` — копіює існуючий віддалений репозиторій на локальний комп’ютер.

### 2. Staging Area та Commits

**Що таке Staging Area і навіщо вона потрібна?**  
Staging Area — проміжна зона перед комітом. Дозволяє вибирати, які зміни увійдуть у наступний коміт.

**Яка різниця між git add . та git add -A?**  
`git add .` додає всі зміни у поточній директорії.  
`git add -A` додає всі зміни у всьому репозиторії (включно з видаленими файлами).

**Як написати гарне commit message?**  
Коротко, ясно, у теперішньому часі, описує зміни, наприклад: `Add calculator function`.

**Як виправити останній коміт?**  
```bash
git commit --amend
3. Гілки та злиття

Що таке гілка в Git і чому вони "легковажні"?
Гілка — це вказівник на коміт. Легковажні, бо створюються швидко і не копіюють всі файли, а тільки посилання.

Яка різниця між git merge та git rebase?
merge об’єднує історії гілок, зберігаючи всі коміти.
rebase переписує історію, переносячи коміти на іншу гілку.

Що таке fast-forward merge?
Злиття, коли цільова гілка не має власних комітів — просто пересуваємо вказівник на останній коміт іншої гілки.

Коли виникають merge конфлікти і як їх уникнути?
Конфлікти виникають, коли змінені одні й ті ж рядки у двох гілках.
Уникати — часто робити git pull, комунікувати в команді, розділяти завдання.

Чому не можна робити rebase публічних комітів?
Бо змінюється історія, інші користувачі отримають конфлікти при синхронізації.

4. Віддалені репозиторії

Що таке origin?
Назва віддаленого репозиторію за замовчуванням.

Яка різниця між git fetch та git pull?
fetch — завантажує зміни з віддаленого репозиторію, не зливаючи їх.
pull — завантажує та автоматично зливає.

Що робить git push --force і чому це небезпечно?
Примусово перезаписує історію на віддаленому репозиторії. Може стерти чужі коміти.

Як видалити віддалену гілку?
git push origin --delete branch_name

5. Скасування змін

Яка різниця між git reset --soft, --mixed та --hard?
--soft — залишає зміни в staging.
--mixed — залишає зміни в робочій директорії, прибирає з staging.
--hard — прибирає зміни з робочої директорії і staging.

Яка різниця між git reset та git revert?
reset змінює історію локально.
revert створює новий коміт, який відміняє попередній, без зміни історії.

Як скасувати git add (видалити файл зі staging)?
git reset HEAD filename

Як повернути видалений файл?
git checkout -- filename

6. Безпека

Що не повинно потрапляти в Git репозиторій?
Паролі, ключі, node_modules, великі бінарні файли без потреби.

Що робити якщо випадково закомітили пароль?
Видалити з історії (git filter-branch або BFG Repo Cleaner) і змінити пароль.

HTTPS vs SSH для GitHub — що краще?
SSH безпечніше та зручніше для частих пушів. HTTPS простіше для одноразового доступу.

Що таке Personal Access Token і навіщо він потрібен?
Токен для автентифікації замість пароля при роботі з GitHub через HTTPS.

7. GitHub

Що таке Pull Request?
Запит на злиття змін з однієї гілки в іншу у віддаленому репозиторії.

Яка різниця між Issues та Pull Requests?
Issue — задача/баг/запит. Pull Request — пропозиція змін коду.

Що таке Fork і навіщо він потрібен?
Копія чужого репозиторію у твоєму обліковому записі, щоб робити власні зміни.

Що таке GitHub Actions?
Автоматизація CI/CD: тестування, деплой, скрипти при комітах або PR.

8. Best Practices

Як часто треба робити коміти?
Часто, після логічних змін. Краще маленькі атомарні коміти.

Скільки часу має жити feature branch?
Поки робота над фічею активна; після злиття можна видаляти.

Як організувати git workflow в команді?
Наприклад: main/master для стабільного коду, develop для інтеграції, feature-branch для фіч.

Чи треба комітити node_modules або інші залежності?
Ні, зазвичай додають у .gitignore та використовують package.json або requirements.txt.
